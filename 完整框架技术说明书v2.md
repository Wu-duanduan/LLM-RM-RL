## 1. 系统定位与总体目标

### 1.1 核心定位

本系统面向**复杂多实体任务场景**，目标是在保持人类语义可解释性的前提下，将高层任务描述稳定地转化为机器人可执行的决策接口，并在环境持续反馈下实现快速重评估与快速调整。

系统的设计重点不在于直接输出底层连续控制，而在于构造一条从**"语义—逻辑—实体—动作约束"**的中间表示链，使上层理解能力与下层决策速度能够有效解耦。

### 1.2 总体架构

## 2. 模块 A：语义任务生成与逻辑对象构造

### 2.1 用途

模块 A 是系统的**慢层语义入口**。它接收人类任务描述、场景上下文和必要的先验规则，输出能够支撑后续在线逻辑编译的高层逻辑对象。

### 2.2 核心职责

1. 将自然语言任务转化为**任务级逻辑对象**
2. 构造**任务推进逻辑**（TRM - Task Relationship Memory）
3. 生成**结构约束对象**（SRM - Spatial Relationship Memory）

### 2.3 输入与输出

| 类别 | 内容 | 说明 |
|------|------|------|
| **输入** | human_command | 人类任务语义 |
| **输入** | global_context | 当前场景描述 |
| **输入** | task configuration | 任务模板或规则约束 |
| **中间表示** | candidate traces | 候选任务轨迹 |
| **中间表示** | LTL formulas | 线性时序逻辑公式 |
| **中间表示** | Bayesian chains | 贝叶斯链条 |
| **输出** | TRM_full | 完整任务奖励机（显式状态机） |
| **输出** | SRM_constraints | 高层事件约束对象 |
| **输出** | TRM_init_state | 当前初始化状态 |

## 3. 模块 B：在线逻辑编译、约束监视与动作空间收缩

### 3.1 用途

模块 B 是系统的**高频在线层**。它接收模块 A 生成的任务逻辑对象，并结合当前物理状态、语义实体状态和局部观测，在每一轮在线循环中维护当前任务状态、结构约束状态和实体合法性。

### 3.2 核心职责

1. 维护 TRM 当前状态（状态机推进）
2. 监视 SRM 约束状态（约束检查）
3. 筛选合法实体（实体级过滤）
4. 生成动作掩码（动作空间收缩）
5. 计算稠密奖励（训练信号）

### 3.3 输入与输出

| 类别 | 内容 | 说明 |
|------|------|------|
| **输入** | TRM_full / curr_trm_state | 完整任务奖励机及其当前运行状态 |
| **输入** | SRM_constraints / curr_belief_state | 结构约束监视器及其当前约束状态 |
| **输入** | physical_states / semantic_map / active_ids | 当前机器人状态、实体状态、活跃实体编号 |
| **输出** | u_T / u_S | 任务状态表示 / 结构约束状态表示 |
| **输出** | active_entities / entity_list_i | 当前被激活且对机器人 i 合法的实体候选集合 |
| **输出** | dense_reward | 稠密奖励 |
| **输出** | hard_mask | 安全性约束掩码（float 0/1），0=禁止，网络内部用 `masked_fill(-inf)` 实施硬屏蔽 |
| **输出** | soft_task_mask | 任务优先级归一化值（priority/10000，连续 0~1），供调用方加权参考；当前版本 PointerDecisionNetwork 不直接消费此字段，由 MARLObservationWrapper 生成后随观测一并传出 |

## 4. 模块 C：实体指针决策模块

### 4.1 用途

模块 C 面向**动态实体集合**进行指针式选择，其职责是在当前合法实体空间中确定"此时此刻应优先指向哪个目标实体或目标事件"。

### 4.2 核心职责

1. 在合法实体空间中进行指针式选择
2. 消费模块 B 给出的结构化状态和过滤结果
3. 输出实体指针和置信度得分
4. 支持 Gurobi 专家模式和 MARL 控制模式

### 4.3 网络架构

**实现**：[`marl_pointer_network.py:PointerDecisionNetwork`](marl_pointer_network.py:16)

```python
class PointerDecisionNetwork(nn.Module):
    """
    实体指针决策网络
    输入: (self_state, u_T, u_S, entity_list, entity_mask, hard_mask)
    输出: pointer_logits [B,N], pointer_probs [B,N], 可选 value [B,1]
    """
    def __init__(self, config):
        super().__init__()
        
        # 1. 自身状态编码器（两层 MLP，hidden=128）
        self.self_encoder = nn.Sequential(
            nn.Linear(4, 128),   # 输入: [vx, vy, water_inv, medkit_inv]
            nn.ReLU(),
            nn.Linear(128, 128)
        )
        
        # 2. 全局状态编码器 (u_T[16] + u_S[8] = 24维输入)
        self.global_encoder = nn.Sequential(
            nn.Linear(24, 128),
            nn.ReLU(),
            nn.Linear(128, 128)
        )
        
        # 3. 实体编码器（14维输入，两层 MLP）
        self.entity_encoder = nn.Sequential(
            nn.Linear(14, 128),  # rel_x(1)+rel_y(1)+类别one-hot(4)+优先级(1)+动作one-hot(7)
            nn.ReLU(),
            nn.Linear(128, 128)
        )
        
        # 4. 交叉注意力机制（Pointer Network 核心结构）
        #    query = self_feat + global_feat  [B,1,128]
        #    key = value = entity_feat        [B,N,128]
        self.attention = nn.MultiheadAttention(
            embed_dim=128, num_heads=4, batch_first=True
        )
        
        # 5. 指针输出层（对 entity_feat 逐实体打分）
        self.pointer_head = nn.Linear(128, 1)
        
        # 6. Value Head（PPO Critic，两层 MLP）
        self.value_head = nn.Sequential(
            nn.Linear(128, 128),
            nn.ReLU(),
            nn.Linear(128, 1)
        )
    
    def forward(self, self_state, u_T, u_S, entity_list, entity_mask, hard_mask, return_value=False):
        B, N, _ = entity_list.shape
        
        # 1. 编码自身状态和全局状态
        self_feat = self.self_encoder(self_state)                           # [B, 128]
        global_feat = self.global_encoder(torch.cat([u_T, u_S], dim=-1))    # [B, 128]
        
        # 2. 编码实体列表
        entity_feat = self.entity_encoder(entity_list)                      # [B, N, 128]
        
        # 3. 融合 query（加法，非拼接）
        query = (self_feat + global_feat).unsqueeze(1)                     # [B, 1, 128]
        
        # 4. 交叉注意力（无效实体被 key_padding_mask 屏蔽）
        attn_output, _ = self.attention(
            query, entity_feat, entity_feat,
            key_padding_mask=~entity_mask
        )
        
        # 5. 指针得分（对 entity_feat 直接打分，非 attn_output）
        pointer_logits = self.pointer_head(entity_feat).squeeze(-1)         # [B, N]
        
        # 6. 应用 hard_mask（float，0=禁止 → -inf 填充）
        pointer_logits = pointer_logits.masked_fill(hard_mask == 0, float('-inf'))
        
        # 7. 应用 entity_mask（bool，无效实体 → -inf 填充）
        pointer_logits = pointer_logits.masked_fill(~entity_mask, float('-inf'))
        
        # 8. Softmax 归一化
        pointer_probs = F.softmax(pointer_logits, dim=-1)                  # [B, N]
        
        if return_value:
            value = self.value_head(self_feat + global_feat)                # [B, 1]
            return pointer_logits, pointer_probs, value
        return pointer_logits, pointer_probs
```

## 5. 模块间接口与运行时序

### 5.1 模块间接口定义

| 接口方向 | 核心对象 | 主要字段 | 作用 |
|----------|----------|----------|------|
| **A → B** | TRM_full | states, transitions, init_state, accept_states | 提供任务推进逻辑 |
| **A → B** | SRM_constraints | event-level constraints, monitors, templates | 提供高层结构约束 |
| **B → C** | u_T, u_S | current task state, current constraint state | 提供当前在线状态表示 |
| **B → C** | entity_list_i, entity_mapping | legal entities for robot i | 提供实体级决策空间 |
| **B → C** | hard_mask, soft_task_mask | 安全约束掩码 / 任务优先级权重 | 提供动作空间收缩结果 |

#### （附）实体特征编码规范

**单实体特征向量（entity_feat_dim = 14）**：

| 偏移 | 维度 | 编码方式 | 说明 |
|------|------|---------|------|
| [0] | 1 | float | 相对坐标 rel_x（目标 x - 机器人 x） |
| [1] | 1 | float | 相对坐标 rel_y（目标 y - 机器人 y） |
| [2] | 4 | one-hot | 实体类别：fire(0), survivor(1), hazard(2), station(3) |
| [6] | 1 | float | 归一化优先级（priority / 10000，截断到 [0, 1]） |

**实体列表张量**：`entity_tensor` shape = `[max_entities, 14]`，不足部分 zero-padding，配合 `entity_mask`（bool）标识有效行。

### 5.2 运行时序

```
初始化阶段:
  模块 A 生成 TRM_full 和 SRM_constraints
    ↓
  下发给模块 B

在线循环:
  模块 B 维护当前状态 (u_T, u_S)
    ↓
  模块 B 筛选合法实体 (entity_list_i)
    ↓
  模块 B 生成动作掩码 (hard_mask, soft_task_mask)
    ↓
  模块 C 进行指针决策 (action_pointer)
    ↓
  环境执行 → 观测更新 → 回到模块 B

重规划触发:
  任务目标重定义 / 场景显著变化
    ↓
  模块 A 重新生成逻辑对象
    ↓
  下发给模块 B
```

```
┌─────────────────────────────────────────────────────┐

│  ↓                                                  │

│  ↓                                                  │

└─────────────────────────────────────────────────────┘
```

## 7. MARL 控制模式

### 7.1 控制模式概述

系统支持两种控制模式：

| 模式 | 决策来源 | 用途 | use_marl_control |
|------|----------|------|------------------|
| **Gurobi 专家模式** | Gurobi 优化器 | 数据收集、专家示范 | False |
| **MARL 控制模式** | MARL 策略网络 | 在线训练、实际部署 | True |

### 7.2 MARL 控制模式实现

**核心类**：[`disaster_env.py:DisasterEnv`](disaster_env.py:992)

```python
class DisasterEnv(Go2Env):
    def __init__(self, num_envs, env_cfg, obs_cfg, reward_cfg, command_cfg, 
                 show_viewer=False, 
                 use_marl_control=False):  # 🔴 MARL 控制标志
        
        # MARL 控制模式
        self.use_marl_control = use_marl_control
        self.marl_task_assignments = {}  # {agent_name: task_id}
        self.marl_last_assignments = {}
        
        if self.use_marl_control:
            print("🤖 MARL 控制模式已启用 - Gurobi 将被禁用")
```

**任务管理方法**：

```python
def set_marl_task(self, agent_name: str, task_id: str):
    """设置 MARL 选择的任务"""
    self.marl_task_assignments[agent_name] = task_id

def get_marl_task(self, agent_name: str):
    """获取 MARL 设置的任务"""
    return self.marl_task_assignments.get(agent_name, None)

def clear_marl_task(self, agent_name: str):
    """清除任务"""
    if agent_name in self.marl_task_assignments:
        del self.marl_task_assignments[agent_name]
```

## 8. 技术要点与实现细节

### 8.1 当前版本的技术要点

1. **显式分层**：模块 A（慢速语义生成）→ 模块 B（高频在线编译）→ 模块 C（实体指针决策）

2. **TRM 是任务推进逻辑的主要承载对象**，适合由轨迹生成、LTL 归纳、贝叶斯链条发现和奖励机构造链路获得

3. **SRM 用于表达高层事件合法性、前置条件与结构约束**，可由 LTL 约束、规则模板或等价监视器对象生成

4. **MARL 控制模式**：支持从 Gurobi 专家到 MARL 策略的无缝切换

5. **系统不要求慢层逻辑对象直接输出精确底层动作**，而是通过模块 B 的在线编译和空间收缩机制，将高层逻辑稳健地转写为可学习、可执行的中间接口
